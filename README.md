# asc-mcp — App Store Connect MCP server

An MCP (Model Context Protocol) server, written in Go, that wraps the
**App Store Connect API** so an LLM agent (e.g. Claude) can automate App Store
metadata: product info, localizations, IAPs/subscriptions, screenshots, and
submitting a version for review — without manual steps in the web console.

Transport: **stdio**. Implementation language: **Go 1.25+**.

---

## What it does

In one conversation an agent can create a version, fill localizations on
ru/uz/en, upload screenshots, attach the version to a review submission, and
submit it for review. Mutating operations are idempotent and recorded in an
audit log.

### Tools (21)

| Group | Tools |
|---|---|
| Discovery | `list_apps`, `get_app_info`, `list_versions`, `get_version_localizations` |
| Metadata / localization | `update_app_info_localization`, `update_version_localization`, `bulk_update_version_localizations`, `create_version_localization` |
| Versions | `create_version`, `update_version` |
| IAP / subscriptions | `list_iaps`, `create_iap`, `update_iap_localization`, `update_subscription_localization` |
| Screenshots | `upload_screenshot`, `list_screenshot_sets`, `delete_screenshot` |
| Submission | `create_review_submission`, `add_version_to_submission`, `submit_for_review`, `get_submission_status` |

## Architecture

Feature-Sliced layout (sliced by domain, not by technical layer):

```
cmd/server                 entrypoint, DI, MCP tool registration
internal/shared
  asc                      generated oapi-codegen client + JSON:API facade
  auth                     ES256 JWT provider (thread-safe, auto-refresh)
  config                   env config + validation (secrets never logged)
  db                       Postgres store (sqlx + squirrel) + goose migrate
  store                    persistence interface + idempotency hashing + Noop
  mcperr                   ASC error -> normalized, agent-friendly error
  validate                 domain limits & editable-state checks (spec §8)
  toolkit                  MCP result helpers + idempotency/dry-run wrapper
internal/features
  apps | versions | localization | iap | screenshots | submission
    handler.go             MCP tools
    service.go             business logic (owns its ascClient interface)
    mocks/                 gomock-generated mocks
internal/platform/http     optional echo server (healthz, metrics)
migrations                 goose SQL (embedded, applied on startup)
api                        openapi.json (full), openapi.pruned.json, codegen cfg
scripts/prune_spec.py      narrows the Apple spec to used operations
```

Within a feature: `handler (MCP tool) -> service -> asc facade / store`.
Each service declares the small `ascClient` interface it needs (consumer-side),
so it is unit-tested against a generated mock.

### The ASC client

The typed client is generated by **oapi-codegen** from Apple's official OpenAPI
spec (v4.3.1). The full spec is ~7 MB / 900+ paths / ~3000 schemas, which is
unusable to generate wholesale, so `scripts/prune_spec.py`:

1. keeps only the ~40 operations these tools call (`scripts/include_ops.txt`),
2. keeps every component reachable from them via `$ref`,
3. strips `enum` constraints (Apple defines colliding top-level + inline enums
   that oapi-codegen cannot name; domain constraints are enforced in our own
   `validate` package instead — spec §8).

A thin hand-written facade (`internal/shared/asc/facade.go`) wraps the generated
client with JSON:API-aware methods; write payloads are sent through the
generated `*WithBody` methods.

## Setup

You need three things from App Store Connect (**Users and Access →
Integrations → App Store Connect API**, generate a key with the *App Manager*
role):

| Value | Where |
|---|---|
| `ASC_ISSUER_ID` | Issuer ID at the top of the API keys page |
| `ASC_KEY_ID` | Key ID of the key you generate |
| the `.p8` file | "Download API Key" — **downloadable only once** |

Run `asc-mcp -setup` at any time to print these instructions.

### Store credentials (recommended)

The server auto-loads a config file, so you set this up once:

```bash
mkdir -p ~/.config/asc-mcp
mv ~/Downloads/AuthKey_*.p8 ~/.config/asc-mcp/ && chmod 600 ~/.config/asc-mcp/AuthKey_*.p8
cat > ~/.config/asc-mcp/config.env <<'EOF'
ASC_ISSUER_ID=00000000-0000-0000-0000-000000000000
ASC_KEY_ID=XXXXXXXXXX
ASC_KEY_PATH=/Users/you/.config/asc-mcp/AuthKey_XXXXXXXXXX.p8
EOF
chmod 600 ~/.config/asc-mcp/config.env
```

Config file search order (first found wins; real env vars override the file):

1. `$ASC_ENV_FILE` &nbsp; 2. `./.env` &nbsp; 3. `$XDG_CONFIG_HOME/asc-mcp/config.env`
&nbsp; 4. `~/.config/asc-mcp/config.env` &nbsp; 5. `~/.asc-mcp/.env`

### Multiple accounts / teams

An API key belongs to one team, so each account = its own credentials. Two ways:

**Profiles** — all accounts stored centrally in `~/.config/asc-mcp/`, each named
and selectable. Create one per account (this copies the `.p8` and writes the
config for you; omit flags to be prompted):

```bash
asc-mcp -add-profile acme     -issuer <ID> -key-id <KID> -p8 ~/Downloads/AuthKey_ACME.p8
asc-mcp -add-profile personal -issuer <ID> -key-id <KID> -p8 ~/Downloads/AuthKey_PERS.p8
```

Then select one:

```bash
asc-mcp -profile acme            # or: ASC_PROFILE=acme asc-mcp
asc-mcp -list-profiles           # show configured profiles
```

**Separate MCP servers** (recommended for MCP clients) — register the server
under different names, each with its own profile:

```json
{
  "mcpServers": {
    "asc-acme":     { "command": "asc-mcp", "env": { "ASC_PROFILE": "acme" } },
    "asc-personal": { "command": "asc-mcp", "env": { "ASC_PROFILE": "personal" } }
  }
}
```

The agent then picks tools by server name (`asc-acme` / `asc-personal`).

### All settings

| Var | Purpose |
|---|---|
| `ASC_ISSUER_ID` | Issuer ID |
| `ASC_KEY_ID` | Key ID |
| `ASC_KEY_PATH` | Path to the `.p8` file |
| `ASC_PRIVATE_KEY` | Alternative: `.p8` contents (PEM; literal `\n` allowed) |
| `ASC_BASE_URL` | Override ASC base URL (optional) |
| `ASC_ENV_FILE` | Explicit config-file path (optional) |
| `ASC_PROFILE` | Select `<profile>.env` for multi-account setups (optional) |
| `DB_ENABLED` / `DB_DSN` | Enable Postgres persistence |
| `ASC_DRY_RUN` | Validate + diff without writing |
| `HTTP_ENABLED` / `HTTP_ADDR` | Optional echo healthz/metrics server |

The private key and minted tokens are **never logged or returned** in tool
responses. Logs go to stderr (stdout is reserved for the MCP stdio transport).
Never commit the `.p8` or `config.env` (both are git-ignored).

## Persistence (optional)

A local Postgres database provides:

- **Idempotency** — a hash of `(tool + normalized input)` maps to a prior
  result, so repeating an `update_*` / `upload_screenshot` is a no-op.
- **Audit log** — who/what/when (`operation_log`).
- **Metadata cache** — snapshots for dry-run/diff.

The layer is isolated behind the `store.Store` interface; with `DB_ENABLED=false`
(or an unreachable DB) the server uses a no-op store and the MCP core keeps
working. Migrations are embedded and applied automatically on startup.

## Build & run

```bash
make build            # -> bin/asc-mcp
make test             # go test -race ./...
make cover            # coverage
make vet

# regenerate the client + mocks (needs python3 for the prune step)
make spec             # download the latest Apple spec (community mirror)
make generate         # prune + oapi-codegen + mockgen

# first-run help: where to get credentials and how to store them
./bin/asc-mcp -setup
./bin/asc-mcp -version

# run over stdio (auto-loads ~/.config/asc-mcp/config.env if present)
./bin/asc-mcp
```

### Register with an MCP client

```json
{
  "mcpServers": {
    "asc": {
      "command": "/absolute/path/to/bin/asc-mcp",
      "env": {
        "ASC_ISSUER_ID": "…",
        "ASC_KEY_ID": "…",
        "ASC_KEY_PATH": "/absolute/path/AuthKey_XXXX.p8"
      }
    }
  }
}
```

## Testing

- **Unit:** services with gomock-mocked ASC clients; shared packages (auth,
  validate, mcperr, store, config, payloads).
- **Domain limits:** table-driven tests on field-length boundaries.
- Run with the race detector: `go test -race ./...`.

Integration tests against an ASC sandbox app are intended to live behind a build
tag and a separate CI job.

## Error handling

ASC error responses (`errors[].code/detail/title`) are parsed into a normalized
structure classified as `auth` (401/403), `validation` (409/422 — field not
editable in the current version state), `rate_limit` (429, with `Retry-After`),
`not_found` (404), `server` (5xx) or `client`. Tools never swallow errors; they
return a structured error block.

## Notes / deviations from the spec

- **Enum stripping** in the generated client (see above) — Apple's spec has enum
  naming collisions that block codegen; validation is enforced in `validate`.
- The optional `/metrics` endpoint serves basic Go runtime stats as JSON rather
  than Prometheus format, to avoid an extra dependency.
- Version-state editability is validated when the caller supplies state (via
  `list_versions`); otherwise it is deferred to Apple, which returns a clearly
  normalized validation error.
